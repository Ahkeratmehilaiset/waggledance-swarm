# 2026-04-27: Atomic flip analysis — flip is a no-op for v3.6.0

## TL;DR

The atomic flip described in `docs/atomic_flip_prep/HUMAN_APPROVAL.yaml` rationale was signed in good faith but was based on a mental model that does not match the actual repo state. After investigation, the conclusion is:

**For v3.6.0, the atomic flip is a no-op.** No filesystem rotation, no `os.replace()`, no runtime pointer change is needed. The work the rationale describes belongs to a future Stage 2 cutover that has not been scoped or implemented yet.

The signed approval artifact is preserved in place for audit, with a SUPERSEDED note pointing to this analysis. Neither this branch nor any subsequent session attempts to execute a flip under that approval.

## What the rationale claimed

From `HUMAN_APPROVAL.yaml` rationale paragraph "SCOPE OF THE CHANGE":

> Atomic flip rotates the runtime read path from legacy locations
> (data/faiss_staging/, data/faiss_delta_ledger/) to the new
> data/vector/<cell>/current/ structure produced by the Stage 1+2
> MAGMA migration. The flip itself is one os.replace() per cell across
> 8 cells, recorded as previous_main_tip_sha before any rotation.

This describes a filesystem-level pointer rotation across 8 hex cells with a `current/` directory per cell.

## What the repo actually has

### Three different storage paths exist

| Path | Purpose | Read by runtime? |
|---|---|---|
| `data/faiss/` | `_DEFAULT_FAISS_DIR` in `core/faiss_store.py` line 26 — runtime's actual read path | **YES** |
| `data/faiss_staging/` | Phase 8 hex-cell staging area (5 of 8 cells present) | NO — only `tools/backfill_axioms_to_hex.py` and `tests/gates/test_disaster_recovery.py` reference it |
| `data/faiss_delta_ledger/` | Phase 8 delta-ledger (107 entries) | NO — same scope as staging |
| `data/vector/` | Stage-1 additive snapshot generated by `tools/migrate_to_vector_root.py` on 2026-04-24 | NO — explicitly Stage-1 snapshot per `docs/architecture/MAGMA_FAISS_SCALING.md §0` |

`data/faiss/` contains entirely different scope than the hex-cell stagings: `agent_knowledge`, `axioms`, `bee_knowledge`, `training_pairs` — none of these are hex-cell directories.

### `current/` symlinks/directories do not exist

For all 8 cells under `data/vector/<cell>/`, no `current/` symlink or directory exists. The rationale's claim of "8 cells current pointers" has no target.

### No `events.jsonl`

`data/vector/events.jsonl`, referenced in the post-flip verification step #6, does not exist. The `vector_events.py` module defines event NAMES (a contract) but per `MAGMA_FAISS_SCALING.md §0` "**No runtime producer yet** — the contract is in place so code paths can emit without a format break later."

### Source-grep results

- `grep -rn "data/faiss_staging\|data/faiss_delta_ledger" waggledance/`: zero matches in runtime code (only tests + tools)
- `grep -rn "data/vector" waggledance/`: only two matches in `vector_events.py` (the contract module), no actual runtime read or write

### `validate_cutover.py` reports cutover already done

```
============================================================
FULL AUTONOMY MODE ENABLED — cutover valmis
============================================================
```

But this is a different cutover: it's the **autonomy runtime cutover** (Phase 1-9 module activation, primary=waggledance, compatibility_mode=false). Not the FAISS hex-cell cutover described in the rationale.

## Three cutovers, not one

| Cutover | What it is | Status |
|---|---|---|
| **#1 Autonomy runtime cutover** | Activate `waggledance/` package as primary runtime, set `compatibility_mode=false` | ✅ DONE (verified by `validate_cutover.py`) |
| **#2 FAISS hex-cell read-path cutover** | Make runtime read hex-cell FAISS indices from a single canonical location | ⏳ NOT DONE — but also not strictly needed yet because runtime doesn't read hex-cell-FAISS at all yet (verified by source-grep) |
| **#3 Stage-2 durable-bus cutover** | Per `MAGMA_FAISS_SCALING.md §2`, this is when JetStream durable bus + vector-indexer consumer come online and the `vector.commit_applied` event drives index rebuilds | ⏳ NOT DONE — explicitly Stage 2 work, not in scope for v3.6.0 |

The HUMAN_APPROVAL.yaml rationale conflates #2 and #3, and assumes a `current/` symlink mechanism that doesn't exist.

## Why this happened

When I prepared the atomic flip artifacts during the unified release finalization session, I generated rationale text based on the Phase 9 architecture documents (`PROMPT_2_INPUTS_AND_CONTRACTS.md`, etc.). Those documents describe what the eventual Prompt 2 atomic flip *would* do conceptually, but the agent's rationale text concretized that into a specific filesystem-rotation mechanism without verifying that mechanism existed in the code.

The operator signed the approval in good faith based on the rationale text. Neither the operator nor the agent verified the ground truth (the actual `_DEFAULT_FAISS_DIR` value, the absence of `current/` symlinks, the actual scope of `data/faiss/`).

This is exactly the failure mode the constitutional principles are meant to catch: a signed approval for an operation that doesn't match repo reality. The right response is to STOP at execution time, document the discrepancy, and not improvise.

## Why not "fix" the rationale and execute something else

The operator was offered three options:

- **A) Cutover #2 = no-op for v3.6.0; Stage 2 work happens later** ← chosen
- B) Make code change in this session (modify `_DEFAULT_FAISS_DIR`, migrate data) — too invasive, too speculative, would conflict with Stage 2's planned NATS JetStream design
- C) Targeted hybrid retrieval cutover (point HybridRetrievalService at `data/vector/`) — requires architectural decisions about how `data/faiss/` (agent_knowledge, axioms, etc.) coexists with hex-cell vectors

Option A is the honest one: v3.6.0 ships the autonomy fabric scaffold, no FAISS hex-cell runtime reads exist anywhere yet, so there is literally nothing to flip.
