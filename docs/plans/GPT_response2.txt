## Executive verdict

**Approve with minor amendments — do not proceed past Phase A until the amendments are written into the plan and tests.**

The refined plan captures the main architectural corrections well: shadow/candidate/authoritative activation, keyword + centroid cell-assignment honesty, multi-view embeddings, independent oracle sources, negation/question-frame parsing, FAISS staging/RCU, and MAGMA retrieval tracing are all materially present. One wording nuance: `enabled: true` plus `mode: shadow` is technically fine, but operationally easy to misread. Consider renaming or documenting it as:

## PART B — Seven blind spots

### 1. Embedding determinism

**Recommendation: must add before Phase A.**

Do not assume `nomic-embed-text` is bitwise deterministic. Add Phase A test `A.6 embedding determinism`: embed 50 fixed strings 30 times each using batch=1, batch=16, and batch=32; restart Ollama once and repeat. Compute `max_abs_diff`, cosine drift, and rounded-vector hash drift. If raw float bytes differ but cosine drift is < `1e-6`, treat vectors as semantically deterministic but do **not** use raw vector bytes for manifest checksums.

Contingency: manifest checksums should be based on source material and metadata, not raw vector bytes:

```text
manifest_checksum =
  sha256(sorted(canonical_solver_id, view_type, text_hash, embedding_model, embedding_dim))
```

Store vector hashes separately as diagnostic evidence:

```yaml
vector_hash_policy:
  raw_hash: diagnostic_only
  rounded_hash_decimals: 6
  semantic_drift_threshold_cosine: 1e-6
```

### 2. Dream Mode + FAISS staging interaction

**Recommendation: must add before Phase A.**

The current “Dream Mode writes to staging only” rule is not enough. Use an append-only delta ledger:

```text
data/faiss_delta_ledger/
  thermal/
    000001_axon_backfill.jsonl
    000002_dream_mode.jsonl
```

Protocol:

```text
1. Axiom backfill builds base staging from axioms.
2. Dream Mode never writes the staging FAISS index directly.
3. Dream Mode appends candidate docs to delta ledger with sequence number.
4. Manifest builder freezes a high-water mark H.
5. Staging = axioms + all deltas <= H.
6. Validate staging.
7. Promote to live manifest if gates pass.
8. Deltas > H remain pending for next manifest.
```

This prevents Dream Mode and backfill from interleaving writes into the same index.

### 3. In-flight query during manifest swap

**Recommendation: must add before Phase A.**

Use entry-time coherent snapshot, not per-cell live reads. At query start:

```python
snapshot = faiss_registry.current_snapshot()
trace.faiss_manifest_version = snapshot.manifest_version
```

Then every cell search uses `snapshot.cells[cell_id]`, never a fresh registry lookup mid-query. The RCU swap publishes a new snapshot only after all cells are coherent; old snapshots are refcounted or kept alive until in-flight queries finish. This prevents `thermal@v3 + energy@v4` mixed queries.

### 3.5. Concurrent-write-to-cell ordering

**Recommendation: must add before Phase A.**

Use an append-only ledger with monotonic sequence numbers, not first-writer-wins. Every staging input should have:

```yaml
cell_id: thermal
source: axiom_backfill | dream_mode | manual_patch
canonical_solver_id: heating_cost
canonical_hash: ...
parent_manifest_version: ...
seq: 1842
created_at: ...
```

Conflict resolution:

```text
same canonical_hash -> duplicate, no-op
same canonical_solver_id + new hash -> new revision, requires validation
different source same seq -> impossible; allocator bug
manual_patch > axiom_backfill > dream_mode only for conflict adjudication, not silent overwrite
```

Promotion should be deterministic by `(seq, source, canonical_solver_id)`.

### 4. Phase C compute cost on edge hardware

**Recommendation: must add before Phase A, but as execution-scope clarification.**

Phase C should run on the workstation, not Raspberry Pi. Phase C is a validation experiment, not the edge runtime path. On edge, run only a reduced microbench:

```text
edge_phase_c_smoke:
  queries: 50-100
  architectures: keyword + hex only
  assert embedding_hot_path=false
  assert keyword-fast-path rate >= configured threshold
  assert p95 route latency stays within edge SLA
```

The real 1000-query three-way oracle run belongs on the development workstation where repeated embedding calls will not distort the edge deployment promise.

### 5. Disaster recovery from axioms alone

**Recommendation: must add before Phase A.**

Add explicit gate:

```text
B.6 Disaster recovery dry run:
  delete temp faiss_live copy
  rebuild from configs/axioms/*.yaml + delta ledger <= manifest_hwm + embedding model
  compare canonical_solver_ids, source text hashes, view counts, manifest checksum
  run oracle smoke
```

Important nuance: if live FAISS includes Dream Mode case trajectories, then **axioms alone are not enough**. Either define live solver index as axioms-only, or include the Dream delta ledger in the disaster-recovery source of truth. The accurate invariant should be:

```text
FAISS live can be rebuilt from:
  configs/axioms/*.yaml
  + faiss_delta_ledger up to manifest high-water mark
  + embedding model/version
  + backfill config
```

### 6. Embedding cache for repeated queries

**Recommendation: add before Phase C.**

This is cheap and high-leverage. Cache embeddings, not routing decisions:

```text
cache_key = sha256(embedding_model + normalized_query_text)
value = vector
```

Do **not** cache final solver route unless the key also includes `faiss_manifest_version`; otherwise a manifest update would make cached routes stale. For the 400h gauntlet, a query embedding cache will reduce repeated embedding load without changing correctness.

Minimal implementation:

```yaml
embedding_cache:
  enabled: true
  backend: sqlite | disk_jsonl | in_memory_lru
  max_entries: 100_000
  key_fields: [embedding_model, normalized_query_hash]
  trace_field: query_embedding_cache_hit
```

### 7. Capability Surface Index formula

**Recommendation: add but defer to follow-up plan; do not block Phase A.**

The multiplicative formula is useful as a **strict safety-gated score**: if trace completeness is zero, or false solver activation is catastrophic, the whole index should collapse. That is a feature, not a bug.

But the current formula should not be the only dashboard score because a single zero hides progress elsewhere. Use two scores:

```text
CSI_strict =
  multiplicative formula
```

and

```text
CSI_progress =
  weighted/geometric trend score with bounded components
```

Also bound the composite-path term:

```text
bounded_composite =
  min(1.0, log1p(useful_composite_paths_total) / log1p(target_useful_composite_paths))
```

Activation gates should rely on individual metrics, not CSI.
